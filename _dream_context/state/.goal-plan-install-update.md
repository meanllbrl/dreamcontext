# Plan under review — Install/Update/Upgrade hardening (incl. custom-file deletion bug)

## Goal
Detect + fix ALL install/update/upgrade problems. Trigger: `dreamcontext update` offered to DELETE a user's custom `.claude/agents/watchlist-monitor.md` that dreamcontext never installed. dreamcontext must never touch/flag/delete files it didn't install.

## Validation (Phase 6 contract)
Automated: regression test per fixed bug (esp. custom `.claude/agents/<x>.md` never bootstrapped/flagged/deleted, incl. polluted-manifest path) + full `npx vitest run` green + `npm run build`.

## Confirmed mechanism
`bootstrapManifestFromScan` (manifest.ts:224-288) adopts EVERY `.md` in `.claude/agents/`, EVERY subdir under `.claude/skills/` + `.agents/skills/`, EVERY `.toml` in `.codex/agents/` as dreamcontext-owned, tagged `PRE_MANIFEST_VERSION`. Custom `watchlist-monitor.md` is indistinguishable at scan time → adopted (manifest.ts:244-250). Next `update`: `diffManifests` (147-172) puts it in `removed` (in old manifest, absent from fresh install manifest), `isSafeDeletePath` passes (under `.claude/`), `pruneStaleFiles` (update.ts:53-105) offers it for deletion — default YES at the confirm prompt (update.ts:86), deleted outright under `--yes`.

TWO independent fixes needed:
- **A** stops NEW over-adoption (allowlist the scan).
- **B** protects users ALREADY polluted by the shipped bootstrap (their manifest already has the custom file as `pre-manifest`; an allowlist does nothing for them). B is the safety net the polluted-manifest regression test exercises.

**Allowlist source (verified):** dreamcontext ships agents from TWO places:
- Core agents (`agent` kind) from repo-root `agents/` via `findPackageDir('agents')` (install-skill.ts:768): dreamcontext-explore, dreamcontext-initializer, sleep-product, sleep-state, sleep-tasks.
- Pack agents (`pack-agent`) from `catalog.agents[].name` (catalog.json): reviewer, discover-brand, council-*, marketing-*, review-*, biv-*, goal-*.
Allowlist = union of BOTH + core skill dir `dreamcontext` + pack/standalone skill names. `review-coordinator` is in NEITHER (verified) → correct "genuinely-removed" example, but cleanable only via a real-version manifest entry, not a bootstrap entry.

## Acceptance criteria
1. Bootstrap records agent/skill files only if name ∈ known set (T1-T3).
2. knownArtifactNames() sourced from catalog.agents+packs+standalone + repo-root agents/ core names; includes reviewer/goal-planner/sleep-state/dreamcontext-explore/engineering/dreamcontext; excludes watchlist-monitor + review-coordinator (T4).

4. Heuristic-flagged entries re-persisted into new manifest EVERY run (not just on cancel) — never silently dropped (verify via T11 surviving a 2nd update).
5. `update --yes` with custom watchlist-monitor.md present — both no-manifest (bootstrap) AND polluted-manifest paths — leaves file on disk (T10, T11).
6. `upgrade --check` reflects live npm latest when reachable; falls back to cache then "unknown" offline (T12, T13; existing tests pass).
7. `upgrade --refresh` does live check + updates .version-check.json (T14).
8. Full existing suite green.

# Review Round 1 — Resolutions (AUTHORITATIVE; supersede conflicting text above)

All 3 reviewers NEEDS_WORK; core architecture (A allowlist + B partition) confirmed sound. Resolutions:

**R1 (pragmatist P1) — `upgrade --check` source selection (preserve existing tests).** Do NOT chain source VALUES with `??` (an injected `latestVersion: () => null` would fall through to a live npm call and break the offline test). Instead pick the source FUNCTION:
```ts
// --check: choose the first DEFINED source fn, call it ONCE, use its result as-is.
const source = opts?.liveLatest ?? opts?.latestVersion ?? defaultLiveLatest;
const latest = source();            // null → print "unknown"
```
- Existing tests inject only `latestVersion` → `source = latestVersion` → its return (incl. `null`) used directly, no live call. The four existing --check tests pass UNCHANGED.
- T12: inject `liveLatest:()=>'9.9.9'` (+ latestVersion `'1.0.0'`) → source=liveLatest → `9.9.9`. T13: `liveLatest:()=>null` → source=liveLatest → null → "unknown". 
- Production (no injection): source=`defaultLiveLatest` (live `npm view`, validated `/^\d+\.\d+/`, null on failure/timeout). On null, optionally read cache for a fallback line; never throw.

**R2 (pragmatist P2) — REMOVE `--refresh` (Bug D) entirely.** YAGNI + user's "no unnecessary work". The detection-lag is mitigated by R1 (`upgrade --check` now does a live check, so the user can check anytime). Drop the `--refresh` option, AC7, and test T14. (If desired later, the right home is the hook TTL, not a new upgrade flag.)

**R3 (critic C1 + security S1) — unify re-persist as `{ removed, keep }`; caller re-persists `keep` UNCONDITIONALLY.** This is the load-bearing fix; both blocking findings collapse into it.
- `pruneStaleFiles(projectRoot, oldManifest, newManifest, isFirstRun, yes)` returns `{ removed: string[], keep: string[] }` where:
  - `candidates` = `diffManifests().removed` filtered by `isSafeDeletePath` (unsafe paths: warn + skip, as today — not in `removed`, not in `keep`).
  - Partition `candidates` by `oldManifest.files[p]?.version === PRE_MANIFEST_VERSION` → `heuristic` vs `owned`.
  - `heuristic`: NEVER deleted (warn "detected, not removed; review manually").
  - `owned`: if `isFirstRun` → flag-only (existing contract, not deleted). Else prompt (or `yes`): confirmed → `rmSync` → into `removed`; declined/cancelled → not deleted.
  - **`keep = candidates.filter(c => !removed.includes(c))`** — i.e. everything NOT actually deleted: all `heuristic` (always) + `owned` that were first-run/declined/cancelled.
- **Caller (update.ts, replacing the `if (pruneResult.cancelled)` block at ~205-212):** for every path in `keep`, re-insert `oldManifest.files[path]` into `newManifest.files` if absent — **unconditionally** (no `cancelled` guard). Remove the `if (pruneResult.cancelled)` guard.
- This simultaneously: (a) keeps heuristic/custom files flag-only AND tracked forever (S1/B), (b) preserves the existing cancel-invariant — owned stale files the user declined stay tracked so the next `update` re-offers them (S1), (c) lets confirmed owned files (review-coordinator) actually delete (C critic).
- Drop the separate `flagged`/`cancelled` fields from the return contract (the warning print happens inside prune; the caller only needs `removed` + `keep`).

**R4 (test fixes for the above):**
- **T7 (mixed) split into yes AND cancel variants:** T7a `yes:true` → review-coordinator(0.4.2) deleted+absent from keep; watchlist-monitor(pre-manifest) kept. T7b `yes:false`/cancel with isFirstRun=false → NEITHER deleted; BOTH in `keep` (owned-declined re-offered + heuristic protected); assert the written manifest still contains both.
- **T8 → isFirstRun is a separate axis:** rename to "first-run flags owned (no partition delete)"; keep isFirstRun=true asserting nothing deleted. ADD T8b isFirstRun=false + concrete-version owned + yes:true → deleted (proves the partition path, not the first-run early-return).
- **NEW T15 (the S1 regression):** isFirstRun=false, mixed old manifest (owned concrete + heuristic), user CANCELS (yes:false, declines) → assert owned-declined file STILL in written manifest AND heuristic file STILL in written manifest AND nothing deleted on disk.
- Drop T14 (D removed).

**Non-blocking (apply): note in a code comment at the partition site that hand-edited manifests assigning a concrete version to a custom file are outside the threat model.**

Net: architecture unchanged; C scoped to a live --check (no --refresh); the re-persist becomes `keep = candidates − removed`, re-persisted unconditionally — closing both the heuristic-drop and the cancel-drop regressions.
