# Review Round 2 ÔÇö Resolutions (AUTHORITATIVE; supersedes R6's dir-delete text)

Round 2: pragmatist SOLID, critic SOLID, security NEEDS_WORK on 2 points about R6 (guard untestable via public API; R6's "(incl. '.')" comment misstated `safeChildPath`). Both fixed by extracting the delete-bound into a pure, directly-testable helper:

**R8 ÔÇö Extract `resolveSkillDirToRemove` (pure, exported, directly unit-tested).** In `src/lib/install-packs.ts`:
```ts
/**
 * Absolute dir to remove for a pack's skills under one platform, or null if `name`
 * does not resolve to a STRICT child of skillRoot. Returns null for '.', '..', names
 * containing '/', absolute paths, and the skillRoot itself ÔÇö so a caller can never
 * rmSync(skillRoot) or anything at/above it. This is the load-bearing delete bound.
 */
export function resolveSkillDirToRemove(skillRoot: string, name: string): string | null {
  const abs = safeChildPath(skillRoot, name);          // null on '\0', '/'-escape, absolute
  if (!abs || abs === resolve(skillRoot)) return null;  // reject '.' (resolves to skillRoot) and any non-strict-child
  return abs;
}
```
`uninstallPack` uses it for BOTH the empty-dir `rmSync` AND the fallback on-disk walk root (compute once per platform, reuse):
```ts
const skillDirAbs = resolveSkillDirToRemove(skillRoot, name);
if (!skillDirAbs) { /* skip this platform's dir ops + warn */ }
else { /* walk root = skillDirAbs; and after file deletes: rmSync(skillDirAbs, {recursive,force}) */ }
```

**R9 ÔÇö Test A10 hits the guard in isolation (no allowlist in the way).** Because `resolveSkillDirToRemove` is a pure exported function, A10 unit-tests it DIRECTLY ÔÇö it does not go through `uninstallPack`/`loadCatalog`, so the catalog allowlist (`UnknownPackError`) never short-circuits it. A10 asserts:
- `resolveSkillDirToRemove('/tmp/p/.claude/skills', '.')` Ôćĺ `null` (would equal skillRoot)
- `resolveSkillDirToRemove(root, '..')` Ôćĺ `null`
- `resolveSkillDirToRemove(root, 'a/b')` Ôćĺ `null` (slash)
- `resolveSkillDirToRemove(root, '/etc/passwd')` Ôćĺ `null` (absolute)
- `resolveSkillDirToRemove(root, 'engineering')` Ôćĺ `<root>/engineering` (the only accepted shape)
Plus one integration assert in the existing uninstall test: a normal uninstall (`engineering`) DOES remove `<root>/engineering` (proves the happy path still deletes via the helper).

**Correctness note (replaces R6's wrong parenthetical):** `safeChildPath(skillRoot, '.')` returns `skillRoot` (NOT null), because in `safe-path.ts` `target === base` satisfies the non-escape condition. The dot-case is rejected ONLY by the explicit `abs === resolve(skillRoot)` check inside `resolveSkillDirToRemove`. Do not drop that check.

This adds one tiny pure function (security-justified, replaces the inline guard) ÔÇö no feature-scope growth; pragmatist/critic SOLID verdicts from round 2 stand. `safeChildPath` and `resolve` are already imported in the route/lib layer; `resolve` comes from `node:path`.

# Review Round 3 ÔÇö Final fix (security-prescribed one-liner; plan now CONVERGED)

Round 3: security found R8's helper code didn't enforce its own documented slash-rejection (so R9's `'a/b' Ôćĺ null` assertion would fail). The reviewer provided the exact one-line fix and confirmed it resolves the finding. Applied as **R10**:

**R10 ÔÇö `resolveSkillDirToRemove` enforces its documented contract.** Add the guard as the FIRST line, so the helper itself rejects slash/null names (matching the R8 docstring and the R9 `'a/b' Ôćĺ null` test):
```ts
export function resolveSkillDirToRemove(skillRoot: string, name: string): string | null {
  if (name.includes('/') || name.includes('\\0')) return null;   // R10: enforce single-segment
  const abs = safeChildPath(skillRoot, name);                    // null on absolute / escape
  if (!abs || abs === resolve(skillRoot)) return null;           // reject '.' (== skillRoot) and non-strict-child
  return abs;
}
```
