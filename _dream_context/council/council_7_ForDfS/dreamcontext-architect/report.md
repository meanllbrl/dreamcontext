## Round 1 — 2026-04-25

### Executive Summary

Verdict: **YELLOW** — fixable, but not as-is. The plan respects the architecture's grain (CLI-first mutations, install-skill distribution, dry-run safety, read-only dashboard, isolated agents) but strains three core rules and creates one anti-pattern. All issues fixable in a day.

### Position

Adopt the plan with three required changes before commit:

1. **Vault parity** — JSON-first is correct for relational state, but every entity gets a sibling `.md` bridge file with frontmatter so Obsidian + the Brain graph stay first-class. The vault expectation in soul.md ("Obsidian + dashboard read same `.md` files") is non-negotiable.
2. **Hook integration** — snapshot must surface active cohorts and pending kill/scale recs; UserPromptSubmit nudges on unconfirmed recs >24h. The plan is silent on this.
3. **Replace `--preset marketing` on council** with a marketing-owned wrapper. Council's domain-agnostic purity is a load-bearing rule.

### Reasoning

**Rule-by-rule audit (citing soul.md + council/SKILL.md + DEEP-DIVE):**

*Respects:*
- Rule 3 (CLI-first): plan §"Hard rules" line 157 — "Sub-agents call CLI commands; they do not import library code directly." Correct.
- Rule 4 (install-skill distribution): agents under `skill-packs/agents/` (line 107). Correct.
- Rule 6 (council isolation): plan consumes council, doesn't bypass.
- Read-only dashboard (soul "no UI mutation surface"): line 219 explicit. Good.
- Dry-run + PAUSED defaults (lines 154–155): keeps the Tilki battle-tested guardrails.

*Strains (must fix):*
- **Rule 7 (`_dream_context/` is canonical store, Obsidian + dashboard read same .md)** — Plan §"Folder Layout" makes the marketing tree JSON-only. Obsidian graphs nothing. **Fix:** every cohort/campaign/creative/competitor gets a sibling `.md` with frontmatter (id, type, fb_id, status, links). JSON stays canonical for state; .md is the graph anchor. Same pattern tasks already use.
- **Rule 1 (context pre-loaded via hooks, never search-on-demand)** — Plan silent on hooks. Active cohorts invisible to SessionStart. **Fix:** snapshot emits `## Marketing` section (active cohorts, last insights pull, pending recs); UserPromptSubmit nudges on unconfirmed Performance Monitor recommendations >24h, mirroring sleep-debt nudge.
- **Rule 2 (300-line cap on core/knowledge files; rem-sleep enforces)** — `marketing-learnings.md` is append-only across 6 sections forever. Breaches cap within weeks of real ad spend. **Fix:** rem-sleep extracts stale hypothesis-ledger entries (older than current cohort) to `knowledge/marketing-archive-<YYYY-Q>.md` when file exceeds 300 lines. Plan's "standard cycle" (line 101) is too vague.

*Violates (anti-pattern):*
- **`--preset marketing` on council** (lines 241–243). Council/SKILL.md line 56: *"Generate personas specific to this topic. No preset library."* — explicit. Marketing baking presets *into* the council command breaks that purity. **Fix:** thin wrapper `dreamcontext marketing council "<topic>"` lives in marketing CLI, generates 4 personas, then calls `dreamcontext council create` + `agent create` underneath. Council stays domain-agnostic. Sets the convention for every future skill.

**JSON-vs-MD verdict:** JSON-first inside the folder is correct for relational state (line 84–86 reasoning is sound). Required bridge `.md` files: per cohort, per campaign, per creative, per competitor handle, plus a single `runs/index.md` LIFO log (not per-run — that would flood the vault). `marketing-learnings.md` stays in `knowledge/`. Keep `.obsidian/graph.json` color-group sync (line 279).

**Rule for future skills earning a top-level `_dream_context/<domain>/`:** A skill earns top-level placement iff (a) it produces *durable operational state* the user inspects between sessions, not just narrative knowledge; (b) that state has *relational structure* the dashboard graphs (FK joins, lifecycle status); (c) it owns *external side effects* (API calls, money spent, deployments) needing an audit trail. Marketing meets all three. Brand-voice, design, growth, engineering, system-prompts do not — they live in `knowledge/`. Codify in DEEP-DIVE.md.

**Hooks integration deltas:**
1. SessionStart — snapshot Marketing section.
2. UserPromptSubmit — nudge on unconfirmed kill/scale recs >24h.
3. rem-sleep — prune `runs/` >30 days (keep last 100), compact `insights/` snapshots into per-campaign rollups, extract stale learnings sections.
4. PostToolUse, PreToolUse, PreCompact — no change needed.
