- **Federation is read-only live-reference — no copy-sync, ever (2026-06-15, ★★, supersedes copy-based v0.8.0 model)**: dreamcontext federation MUST maintain single-source-of-truth. A 'reads' connection means recall queries live across the peer vault at call time — nothing is ever copied or materialised locally from a peer. The copy-based sync+drain+digest-inbox model (v0.8.0) was reverted after a user correction: "a reads connection COPIES peer content instead of reading it live — fix the read path so it references." sync and drain are now inert no-ops preserved for the roadmap but NOT active. 'federation purge' removes any stale materialised docs. The Excalidraw federation diagram still shows the old push/pull model — it needs regeneration. Do NOT reinstate copy-based federation without explicit instruction; it violates the single-source-of-truth constraint that is a hard requirement for this project. Source: bm_gpXj7w_b; bm_2L91Eqs5; commit:e464190; knowledge:federation-cross-vault; changelog:2026-06-15|federation.
- **Person tags are the single source of truth for task assignment (2026-06-15)**: The scalar 'assignee' field in task frontmatter is legacy and migrates to person tags ('person:<slug>') on read; it is then deleted. Effective assignees = union(person tags, legacy scalar field) filtered to roster. The UI must surface a person picker (not a text field) so users can assign without knowing internal slugs. ClickUp sync sends CREATE with assignees:ids and UPDATE with an {add, rem} delta. The routes/tasks.ts roster merge is slug-based and provider-agnostic (no 'clickup' substring). When removing an assignee via any UI path (chip row or tag row), both the person tag AND the legacy assignee field must be cleared atomically to prevent ghost reappearance. Source: transcript:a84d75bb; file:src/lib/task-backend/clickup.ts; file:dashboard/src/components/tasks/TaskDetailPanel.tsx; changelog:2026-06-15|tasks.
- **ClickUp singleton tag reconciliation: use live PUT-response tags, not base snapshot (2026-06-15)**: Tags that must be single-valued (e.g. version:<v>) MUST reconcile against the live remote tags returned in the PUT response, not the base snapshot. The base snapshot drifts whenever the value is changed via a bound custom FIELD rather than the tag itself (clickup.ts:1078 — field outranks tag). Non-singleton tags continue to diff against the base snapshot (so remote-only tags survive until a pull merge). Pattern: strip all live version:* tags except the desired one; add the desired tag if absent from the live set. Do NOT revert to base-snapshot-only diffing for version tags — it reliably produces duplicate version:x + version:y on the remote. Source: file:src/lib/task-backend/clickup.ts; file:tests/unit/clickup-tags.test.ts; changelog:2026-06-15|clickup.
- **Sleep/goal agent task-status policy: reality-based, not reflexive (2026-06-15, canonical)**: Removed the hard rule that sleep/goal agents always bump touched tasks to in_review. New policy everywhere (agents/sleep-tasks.md, skill/SKILL.md, skill-packs/goal-skill/SKILL.md): use `completed` for done+low-risk+already-validated work (chores, docs, mechanical refactors, tests-green fixes); use `in_review` ONLY when a human genuinely must verify something (user-facing behaviour change, design/architecture decision, risky/critical-path work, or to hand over a close decision on superseded/abandoned tasks). Goal-skill Phase 6 PASS now closes directly — validation passing with evidence IS the definition of done. Test: "would the user actually want to look at this before it's closed?" If yes → in_review; if it's done and nothing to second-guess → completed. When genuinely unsure, prefer in_review. Source: bm_3hEiNKdQ (salience 3); bm_zmFDh5NA (salience 3); file:agents/sleep-tasks.md; file:skill-packs/goal-skill/SKILL.md; changelog:2026-06-15|sleep.
- **Desktop settings that must survive app restarts go in per-project server-side JSON, not localStorage (2026-06-14, reinforced v0.8.3)**: The Tauri desktop app binds a fresh loopback port each launch, making `http://127.0.0.1:<port>` a different localStorage origin every time — localStorage is reset. Pattern: any setting that must persist across launches (e.g. Sleepy enabled, hotkey, brain graph settings) MUST be written to a server-side JSON file (Sleepy → `~/.dreamcontext/sleepy.json`; brain graph settings → `_dream_context/state/.brain-settings.json`) and hydrated into localStorage on mount. useGraphSettings.ts demonstrates the write-through pattern. Apply this to all future desktop-only persistent settings. Source: task:sleepy-notch-capture; commit:a41dc7b; file:src/server/routes/ui-settings.ts; changelog:2026-06-14|launcher.
- **Animated WebP over `<video>` for WKWebView mascot clips (2026-06-14)**: WKWebView blocks video autoplay unconditionally — the `muted` HTML attribute is ignored; WKWebView checks the muted DOM property, and even setting it via ref+play() on mount is insufficient. Fix: serve mascot clips as animated WebP via `GET /api/sleepy/anim?mode=` and render with `<img>` — img autoplays animated WebP unconditionally in WKWebView, no JS needed. Do NOT revert to `<video>` for mascot clips in the desktop app. Source: commit:b30d311; changelog:2026-06-14|sleepy.
- **Desktop user-tool spawn requires -ilc (interactive login shell) (2026-06-14)**: A non-interactive login shell (`-lc`) skips `~/.zshrc` — tools added to PATH there (e.g. `~/.local/bin` where `claude` lives) are invisible in a Finder-launched Tauri app. Use `/bin/zsh -ilc` for any child spawn that must see the user's full terminal PATH. Capture stdout and stderr separately so rc-file init noise never pollutes tool output. Source: commit:1a69f7a; changelog:2026-06-14|sleepy.
- **Launcher federation graph: A reads B = 'out' connection on A's side (2026-06-14)**: The drag-to-connect interaction on the LauncherGraph creates a directed 'reads' arrow from A to B. This maps to `direction: 'out'` on A's connection record (A has an outgoing connection to B), consistent with the federation-recall out/both model established in 0.8.0. Two-way links create an 'out' connection on BOTH sides. This is the canonical translation between the visual metaphor ("reads") and the data model (`direction: out|in|both`). Source: commit:a41dc7b; file:dashboard/src/pages/LauncherGraph.tsx; file:src/server/routes/launcher.ts.
- **Per-project update status: needsUpdate = compareVersions(setupVersion, dreamcontextVersion()) < 0 (2026-06-14)**: Each vault in the launcher carries a status dot: GREEN (setupVersion >= CLI version), YELLOW (setupVersion < CLI version, warranting `dreamcontext update` in that project), RED (folder missing, removable). The version comparison uses the existing `compareVersions` utility from `src/lib/version-check.ts`. The inline Update button calls POST /api/launcher/update which spawns `dreamcontext update` in the project's directory. Source: commit:a41dc7b; file:src/server/routes/launcher.ts.
- **Desktop server endpoints must do deterministic work IN-PROCESS, not shell out to CLI (2026-06-14)**: In a packaged Tauri desktop app, the resolvable CLI binary (resolved via `$SHELL -lc command -v dreamcontext`) can be a stale/older Homebrew global or absent entirely — its PATH and version are unreliable. Server route handlers that need to run dreamcontext operations MUST call the relevant library functions directly (e.g. `insertToJsonArray`, `loadCatalog`, `vaults.*`) rather than spawning a child `dreamcontext <subcommand>` process. The server IS dreamcontext; in-process calls are deterministic, version-safe, and remove an entire failure class. Non-deterministic side effects (e.g. fire-and-forget `claude` enrichment) may still be spawned as child processes, but MUST have an `'error'` event listener so a missing binary cannot crash the server. Source: task:sleepy-notch-capture; file:src/server/routes/launcher.ts; changelog:2026-06-14|sleepy.
- **Desktop cross-launch config: persist server-side, not in localStorage (2026-06-14)**: The Tauri desktop app binds a fresh loopback port each launch, so `http://127.0.0.1:<port>` is a different localStorage origin every time — all localStorage is reset. Any setting that must survive app restarts (e.g. Sleepy enabled, hotkey) MUST be written to a server-side JSON file (e.g. `~/.dreamcontext/sleepy.json`) and seeded into `localStorage` on mount. Never rely on localStorage alone for cross-launch persistence in the desktop app. Reusable pattern for all future desktop-only settings. Source: task:sleepy-notch-capture; file:desktop/src/sleepy.ts.
- **Multi-global-CLI staleness gotcha (2026-06-14)**: The thin-shell Tauri app resolves the global CLI via `$SHELL -lc command -v dreamcontext`. On a machine with BOTH homebrew and nvm node, homebrew resolves first. ensureCliInstalled() may install to homebrew, so the app runs a DIFFERENT (older) global than the one a dev updated via nvm. Real fix: publish to npm so all resolved globals converge to the same version. Know this before debugging "why is the app running old CLI" issues.
- **npm CLI version line and desktop app version line are INDEPENDENT (2026-06-14)**: npm 0.8.x and desktop v0.8.x tags happen to share the same number but are NOT coupled. The desktop release tag is v0.8.1; the npm package was published as 0.8.0 then will be 0.8.1. Same numbers are coincidental. Never assume a version bump to one implies the other.

- **Multi-vault = multi-window, one shared Node server (2026-06-13)**: The Tauri desktop beta uses a single shared Node server in "launcher mode" (contextRoot=null). - **Tauri v2 ACL blocks custom commands on remote-loaded pages — use built-in WebviewWindow API (2026-06-13)**: Tauri v2 treats http://127.0.0.1:PORT as a REMOTE origin. Fix: use the built-in invoke_handler `WebviewWindow` API (capability: `core:webview:allow-create-webview-window`) — built-in commands bypass the remote-URL ACL check. - **CSS Motion Path for SVG flow animations (2026-06-06)**: Animate `offset-distance` on a small dot element along an `offset-path` SVG path, NOT `stroke-dashoffset` on the full path. `stroke-dashoffset` re-rasterizes the entire stroke on the main thread every frame (visible stutter at >10 dots). Motion Path is compositor-class — only a tiny region is dirty per frame; measured worst-frame 9-11ms, 0 jank at ~27 dots. Glow = radial-gradient fill on the dot (rasterized once), not a per-frame drop-shadow filter. Source: dashboard/src/components/about/FlowDiagram.css.
- **`overflow-x: clip` vs `overflow-x: hidden` for sticky descendants (2026-06-06)**: `overflow-x: hidden` establishes a scroll container on that axis (the other axis computes to `auto`), which causes `position: sticky` descendants to resolve to that element as their scroller rather than the real page scroller — breaking the sticky behavior entirely. Use `overflow-x: clip` to clip horizontally without establishing a scroll container. Source: dashboard/src/components/about/AboutPage.css; blocking issue discovered during sticky scrub section implementation.

- **Recall engine: BM25F + stemming + synonyms + score/rank decoupling (2026-06-02)**: recall.ts uses BM25F field weighting (title×3, tags×2, description×2, body×1), TR suffix folding + EN stemming, and a query-time synonym map. This preserves gate thresholds unchanged (BM25 fallback ≥2.0 changed to .some() in Phase-5 to survive reordering; skill gate ≥1.0; explore agent ≥5/<2). Proven on 60-query gold set: recall@1 68→85%, recall@3 82→95%. Source: src/lib/recall.ts; eval/gold.jsonl; commits a56f9d4, e31b2e3.
