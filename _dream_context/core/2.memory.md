### 2026-03-01 - UserPromptSubmit hook: persistent sleep debt reminders

Added `hook user-prompt-submit` subcommand to `src/cli/commands/hook.ts`. Fires on every user message via Claude Code's UserPromptSubmit hook. Reads sleep debt from `.sleep.json` and outputs a one-line reminder when debt >= 4. Silent when debt < 4. Critical (★★★) bookmarks override the debt threshold. Read-only — no writes to state. ## Technical Decisions

- **Root `npm run build` required for dashboard changes**: `dashboard/npm run build` builds only `dashboard/dist/`. The CLI serves from `dist/dashboard/`, which is populated by tsup's `onSuccess` hook during root `npm run build` (runs `build:dashboard && build:cli`). Never claim a dashboard change is live without confirming `dist/dashboard/assets/` timestamps match source. This caused 3 wasted debugging rounds.
- **Council UI: final report is the primary payload, must be above fold**: Three Council UI iterations confirmed this design principle. Any layout where the synthesized final report is buried (in a non-default tab, behind a drawer, below persona lists) will be rejected. Overview tab = final report as hero. Dynamic section parsing required (final-report.md emits `## Verdict` + `## Appendix` reliably, other sections vary by synthesizer output).
- **Council dashboard: Hall grid + Detail pattern, not split-pane**: CouncilHall is a searchable card grid; clicking a card does full-page navigation to CouncilDetail with back button. Split-pane was rejected (sizes inefficient). Full-page detail allows proper vertical space for Overview (long report), Agents tab (searchable transcript), Matrix tab (full grid).
- **install-claude-md is a standalone command, not bundled with install-skill**: CLAUDE.md is a project-root file (not a `.claude/skills/` entry), owned by the user's project. Install-skill manages Claude Code hooks and skills. Keeping them separate avoids `install-skill` silently overwriting user's CLAUDE.md. Idempotency via HTML comment fences (`<!-- dreamcontext:start/end -->`) allows re-runs to update only the dreamcontext block.
- **assertSafeSegment + assertWithinCouncil pattern for user-supplied path segments**: Any CLI path segment derived from user/agent input (debate IDs, persona slugs) must pass two guards: (1) `assertSafeSegment` — rejects traversal chars (`/`, `\`, `\0`, `.`, `..`) before the path is constructed; (2) `assertWithinCouncil` — resolves the full path and asserts it stays under the council root dir (defense-in-depth). Validate at every lookup site, not just at creation. This is the correct pattern for any CLI command that takes external identifiers and uses them to build file paths.
- **Fence-aware markdown section parser**: When splitting a markdown file on `## Section N` headers to find or replace round sections, track triple-backtick fence state. A `## Heading` line inside a fenced code block is content, not a section boundary. Failing to track fences causes data loss when appending/replacing sections in files that include code examples. Pattern: maintain a `bool inFence` toggle on ` ``` ` lines; only treat `##` lines as boundaries when `!inFence`.
- **Council: one generic council-persona.md, not per-archetype files**: Persona identity comes from `context-and-persona.md` on disk (loaded via `council round-context`). The sub-agent file enforces the CLI protocol. Avoids a sprawling library of persona archetypes that would need maintenance and may not fit the topic anyway.
- **Council: main agent reads only executive summaries**: `council summaries` prints only the `### Executive Summary` (≤150 words) sections per persona. Main agent never touches full reports. This keeps orchestrator context under ~20K tokens even for 3 rounds × 6 agents.
- **Council: synthesizer is a separate sub-agent, not main agent**: Main agent calls `council synthesize` (gets manifest), dispatches synthesizer sub-agent with opus, synthesizer reads every report and writes final-report.md. Main agent's context never expands to full report content.
- **Council: nested folder layout is a deliberate convention break**: Standard dreamcontext is flat files under `_dream_context/{core,knowledge,state}/`. Council uses `_dream_context/council/{debate_id}/{persona_slug}/` because N personas × M rounds × researches doesn't scale flat. Document this in skill as an exception, not a pattern to follow elsewhere.
- **Council: promotion is two-path, no forced synchronous decision**: Main agent asks post-synthesis (fast path). rem-sleep agent picks up unpromoted completed debates via `council list --unpromoted` and auto-promotes or flags based on user engagement. Avoids forcing a decision and keeps `knowledge/` curated.
- **Council: model diversity per persona uses Agent tool `model:` param**: Claude Code's Agent tool accepts a `model:` parameter per dispatch. Main agent assigns opus/sonnet/haiku per persona to create genuine cognitive diversity. Not a dreamcontext feature — it's a Claude Code dispatch capability.
- **react-force-graph-2d chosen for dashboard brain graph**: ~60KB gzip, native force-directed layout, React-friendly, supports custom nodes/hover/zoom. Sigma.js (smaller bundle) rejected because WebGL requires more setup; react-flow rejected (not force-directed). cytoscape.js rejected (109KB, overkill for 50-200 nodes).
- **install-skill --packs uses @inquirer/prompts checkbox, not confirmation per pack**: A single multi-select checkbox list shows all available packs at once. User selects all desired packs in one interaction, then sees a summary before copying. Better UX than one confirm prompt per pack (N prompts for N packs). `alwaysApply: true` packs are marked with a badge so users know they will always-load.
- **--list flag reads catalog.json, no file system scan needed**: Pack discovery is fully driven by catalog.json. `--list` reads it and formats the output. No Glob or readdir required. This is the correct pattern for all skill-pack CLI operations.
- **Base install-skill (no --packs) now hints about packs**: After the base skill installs, the CLI prints a one-liner hint that optional skill packs are available via `--packs`. Discovery happens at the natural moment, not buried in docs.
- **skill-packs install flat into .claude/skills/{pack-name}/SKILL.md**: Preserves all existing cross-references in skill content. Agents stay in `skill-packs/agents/` (flat, no subdirs) and install to `.claude/agents/` separately. catalog.json top-level `agents` array links each agent to its pack via `pack` field; packs link back via `relatedAgents`.
- **skill-packs: base skill IS the pack SKILL.md**: No separate base file. `design/SKILL.md` is the always-loaded design-principles content plus sub-skill reference table. Sub-skills are flat .md files, not subdirs (exception: firebase packs with `references/` subdirs).
- **brand-voice guidelines save to _dream_context/core/, not .claude/**: Guidelines file becomes a numbered core file (e.g., `7.brand_voice.md`), frontmatter holds `strictness` and `always_explain` settings. No `.claude/brand-voice.local.md`. Consistent with dreamcontext core-file pattern.
- **Non-alwaysApply skill descriptions must be trigger-specific**: Claude Code uses `description` to decide when to load the skill. Generic descriptions like "growth strategy" don't trigger reliably. Descriptions must list specific keywords/concepts the skill handles (e.g., "D30 retention benchmarks, TikTok distribution, paywall optimization").
- **Versions unified with Releases (no separate VERSIONS.json)**: Planning-stage versions are just ReleaseEntries with `status: 'planning'`. One file, one schema, backward compat (entries without status field are treated as released). Avoid a second VERSIONS.json that diverges from RELEASES.json over time. The distinction is UI-only — snapshot separates them into "Upcoming Versions" vs "Latest Release" sections.
- **Eisenhower Matrix excludes completed tasks**: The quadrant view is a prioritization tool for future work, not a history. One-line filter: completed tasks are dropped before quadrant assignment. Completed work lives in the Kanban "Completed" column view instead.
- **MultiSelectFilter type-ahead threshold is 5**: Search input hidden when <= 5 options (visible, scannable). Appears at 6+ options (list too long to scan). The threshold is a UX constant, not user-configurable. Apply the same pattern to any future multi-select filter with potentially long lists.
- **urgency defaults to 'medium' on readTask()**: New required field, backward compat with existing tasks that lack it. Same pattern used for other optional fields — server-side default, never undefined in the client.
- **sleep_started_at guard suppresses duplicate sleep dispatch**: Both `getConsolidationDirective()` and the `user-prompt-submit` hook check `sleep_started_at` before emitting any consolidation directive. If consolidation is already running, high-debt sessions get a "Do NOT dispatch" message; low-debt sessions get silence. This prevents parallel sleep agents from spawning from the same session.
- **dreamcontext-explore budget caps prevent deep search spirals**: Hard caps per thoroughness level (Quick 1-3, Medium 4-8, Thorough 9-20 tool calls) based on SWE-Search (ICLR 2025) finding that breadth beats depth. Stops agents from spending 20+ tool calls on an exploration that should take 5.
- **Track C (Reusable Component Check) is always-on**: The explore agent proactively outputs a "Reusable" section flagging existing components relevant to the caller's task even when the caller didn't ask. This is the enforcement mechanism for SKILL.md rule #12. Without it, agents forget to check existing implementations.
- **execFileSync over execSync for hook subprocesses**: Shell injection risk eliminated. `execFileSync` takes an array of args; no shell interpolation of file paths. All PostToolUse and PreCompact subprocess calls use this pattern. Never regress to execSync with string interpolation in hook.ts.
- **findProjectConfig() merges walk-up into single pass**: Previously `findFormatterConfig()` and `findTsconfig()` each walked up the directory tree independently (double I/O). `findProjectConfig()` does one walk, collecting all config types. Public functions delegate to it. Direct callers (post-tool-use handler) use the merged result.
- **resolveLocalBin() + npx fallback pattern**: `resolveLocalBin(bin, cwd)` checks `{dir}/node_modules/.bin/{bin}` walking up from cwd. Falls back to npx only if local binary not found. Avoids cold npx bootstrap on every edit in projects that have biome/prettier installed.
- **tsc --incremental flag**: First run ~5 seconds (full type-check). Subsequent runs <1 second (incremental cache). Cache stored in `tsconfig.tsbuildinfo`. Required for PostToolUse to be non-disruptive in large projects.
- **PostToolUse single subcommand for format + tsc**: Combining both into `hook post-tool-use` avoids registering two separate PostToolUse hooks that would each add latency. One hook invocation, sequential format then tsc.
- **compaction_log as audit trail**: PreCompact hook writes records that will be surfaced in the snapshot when implemented. Current state: write-only (intentional). Future: snapshot shows last compaction event so agents know context was compacted mid-session.
- **ensureHooks() data-driven HOOK_SPECS table**: Refactored from 160 lines of copy-paste to a HOOK_SPECS array where each entry is `{name, matcher, timeout, hookType}`. The registration loop is 15 lines. Adding a new hook = one line in HOOK_SPECS. Never regress to manual copy-paste hook registration.
- **UserPromptSubmit hook fires on every user message for persistent reminders**: SessionStart fires once per session — agents can dismiss it. UserPromptSubmit fires each time the user sends a message, so consolidation reminders repeat until acted upon. This is the correct hook for persistent debt awareness. The hook is read-only (no state writes). PostToolUse was rejected: fires mid-work, wrong timing.
- **UserPromptSubmit uses same thresholds as getConsolidationDirective() but single-line**: Full multi-line directives work in SessionStart output (prepended before snapshot). In UserPromptSubmit context, a compact one-line message is better — it will appear inline during the conversation. Same debt levels (4/7/10/critical bookmarks), same logic, condensed format.
- **tasks list excludes completed by default**: Consistent with how `getActiveTaskLines()` in snapshot.ts filters `status === 'completed'`. Default view matches what agents see in auto-loaded context. `--all` flag shows everything; `--status` for specific filter. `validStatuses` includes `new` to handle projects that use non-standard status values.
- **PreToolUse hook strategy for Explorer vs Plan**: Explorer behavior contradicts built-in sub-agent system prompt (lower-priority additionalContext can't override it) — full replacement via deny hook required. Plan behavior is additive (offering task creation doesn't contradict built-in plan behavior) — SubagentStart injection is sufficient. This asymmetry drove different strategies.
- **transcript distill auto-filters by consolidated_at**: When a session has already been through a consolidation cycle, `transcript distill` defaults to showing only content after `consolidated_at` from sleep history. This prevents redundant processing. `--full` overrides. Entries without timestamps always pass through.
- **SleepHistoryEntry stores session_ids for distill filtering**: `session_ids: string[]` in each history entry enables `transcript distill` to find the relevant `consolidated_at` timestamp for auto-filtering without scanning all sessions.
- **additionalContext is lower priority than sub-agent system prompt**: Claude Code injects SubagentStart `additionalContext` at lower priority than the sub-agent's own system prompt. We cannot change this prioritization. The correct fix is making the main agent (Opus, full context) smarter about delegation — explicitly including `_dream_context/` file paths in Explore/Plan prompts. The SKILL.md Sub-Agents section now instructs this directly.
- **freshDefaults() prevents shared-reference mutation**: `DEFAULT_SLEEP_STATE` spread shared array references (bookmarks, triggers, etc.) across calls, causing test pollution. `freshDefaults()` in `sleep.ts` constructs a fresh object each time. Always use it when initializing an empty SleepState.
- **Bookmarks as primary consolidation signal**: Separating waking-phase memory selection (bookmarking) from sleep-phase consolidation is the key neuroscience insight. The sleep agent processes critical (salience 3) bookmarks first, then processes by change count. This mirrors the Joo & Frank 2025 hippocampal model.
- **Trigger fired_count persisted inside generateSnapshot()**: `writeSleepState()` is called within `generateSnapshot()` after matching triggers so fired counts survive without a separate persist step. Triggers expire in `sleep done`.
- **Transcript distillation is structural, not semantic**: Pure Node.js JSONL line filter. No AI call needed to reduce a 50K-line transcript to 200 high-signal lines. Tool name pattern matching is sufficient.
- **Warm knowledge uses 7-day window + tag overlap**: Two signals determine "warm": accessed in the last 7 days (from knowledge_access map) OR tag overlap with currently active tasks. First paragraph only (handles YAML frontmatter). Pinned knowledge remains separate (full content).
- **System flow lives in core/6, not knowledge/**: The lifecycle document is a core reference that the main agent should always be able to find from the snapshot. Knowledge files are for deep research on specific topics.
- **marked over react-markdown for task body rendering**: `marked@^15` is ~14KB gzipped with zero transitive deps. Safe to use `dangerouslySetInnerHTML` because content is local files only, never user-supplied remote content.
- **tool_count scoring uses Math.max over scoreFromChangeCount**: Bash-heavy sessions with no file writes previously scored 0. `scoreFromToolCount` (0/1-15/16-40/41+ → 0/1/2/3) provides a floor without double-counting work that involves both reads and writes.
- **selectedSlug over selectedTask in KanbanBoard**: Store slug string in state, derive Task via useMemo from live query. Prevents stale snapshots after mutations without additional refetch logic.
- **Release discovery is built-in, not deferred**: `releases add --yes` auto-discovers all unreleased items and back-populates features. The richer schema (features[], tasks[], changelog[]) shipped in the same session as discovery.
- **FieldChange net-change detection**: Change tracker folds A->B->C into A->C and cancels A->B->A. Append-only actions and entries without a `fields` array bypass detection.
- **SQL ER diagram uses SVG bezier curves, not a graph library**: D3/React Flow overkill. SVG paths with cubic bezier + arrowheads over CSS grid entity layout.
- **sql-parser.ts uses line-by-line parsing, not comma-split**: Comma-split breaks on multi-line column definitions and inline comments. Line-by-line tracks context (inside table, current field) and handles REFERENCES, JSONB comment annotations, and isSqlType validation without regex soup.
- **SubagentStart hook outputs JSON, SessionStart outputs plain text**: Driven by Claude Code's hook specs, not a design choice. Do not conflate these formats.
- **SubagentStart briefing is lighter than snapshot**: Sub-agents are task-focused and short-lived. Excludes soul/user/memory to avoid leaking internal agent rules. Directive ("BEFORE searching code, read _dream_context/ files") placed at the very top; features section at position 3 with direct Read paths. Bottom-loading context was being ignored by sub-agents.
- **Stop hook is synchronous**: Uses `readFileSync(0, 'utf-8')`. Guarantees .sleep.json is written before the next SessionStart hook fires. Async would create a race condition.
- **Sleep epoch-based clearing**: `sleep start` records a timestamp; `sleep done` only clears records from before the epoch. Post-epoch records survive for the next consolidation cycle.
- **Snapshot: no token budget, no truncation**: More tokens beats forgetting. If context grows too large, fix with better organization (pinning, knowledge extraction), not truncation.
- **ESM-only, no bundled deps**: chalk v5+ and nanoid v5+ are ESM-only. tsup externalizes all runtime deps to avoid ESM/CJS dual-package hazards.
- **Keyword search, not semantic**: `src/lib/search.ts` tokenize + substring match with score weighting. Zero dependencies, adequate for the use case.
- **LIFO default in insertToJsonArray**: `position: 'top'` is the default. CHANGELOG.json and task changelogs depend on this — do not change without auditing all callers.
- **Snapshot is chalk-free**: Consumed by the SessionStart shell hook. ANSI codes break shell hook processing.

## Known Issues

- **1 pre-existing flaky test**: `tests/unit/id.test.ts` (`generates 8-char suffix after prefix`, nanoid(8) intermittently returns 5 chars). 490 tests total, all passing in normal runs, but the flaky test may still intermittently fail. Fix nanoid usage or mock before next publish.
- Integration tests require a manual `npm run build` before they can run against `dist/index.js`. `npm run build` builds dashboard first, then CLI.
- `getTemplateDir()` in `init.ts` uses a candidate-path approach. Unusual pnpm node_modules layouts could cause silent template resolution failure and produce placeholder stubs.
- The interactive mode (readline REPL) creates a fresh Commander program for every command. Slightly wasteful but not a problem at current scale.
