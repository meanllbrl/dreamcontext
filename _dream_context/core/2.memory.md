- **"Unregistered version" is a ghost — not a deletable RELEASES.json record (2026-06-29)**: A version string that appears only on tasks' `version:` frontmatter field with no matching RELEASES.json entry is called a "ghost." The dashboard VersionsPopover derives ghosts live from task frontmatter; they have no record to delete. To remove a ghost: re-point or clear the referencing tasks' `version:` field. To "register" a ghost: add a RELEASES.json entry via `dreamcontext core releases add`. The new PATCH-rename and DELETE routes handle ghosts gracefully (rename is a no-op on the absent entry; delete clears tasks only). Never try to delete a ghost by editing RELEASES.json — the entry does not exist there. Source: changelog:2026-06-29|dashboard; file:dashboard/src/components/tasks/VersionsPopover.tsx; file:src/server/routes/changelog.ts.
- **Desktop release CI derives VERSION from the git tag, not tauri.conf.json — keep both in sync (2026-06-29)**: `.github/workflows/desktop-release.yml` runs on a pushed `vX.Y.Z` git tag (macos-14 runner → tauri build → ad-hoc deep-sign → `.app.tar.gz` + `.sha256` → GitHub Release). The workflow extracts the version as `VERSION="${TAG#v}"` from the git tag, not from `tauri.conf.json`. However, `tauri.conf.json` still controls the `.app`'s internal version string (displayed in About and used by Sparkle update checks) — if it is stale the .app reports the wrong version even though the release tag and npm package are correct. Rule: whenever bumping the version for a release, update `package.json`, `tauri.conf.json`, and `Cargo.toml` together, then push the tag. npm publish is a separate manual step (not automated by CI). Source: commit:6638cb5; file:desktop/src-tauri/tauri.conf.json; file:.github/workflows/desktop-release.yml; changelog:2026-06-29|release.

- **Desktop app dev loop: `npm run build` + ⌘Q+reopen, never `tauri build` for dashboard/CSS/React changes (2026-06-28, recurring)**: The Tauri app spawns the global CLI at launch time and serveStatic reads from disk per request (index.html no-cache; assets are immutable-hashed). A plain ⌘R reload hits WKWebView's document cache on the SAME loopback port — stale. A full ⌘Q + reopen picks a NEW random port each launch, so the origin changes, the WKWebView cache is bypassed, and the freshly-built dist is served. Fast loop: `npm run build` (root, not just `build:dashboard`) + ⌘Q+reopen. `tauri build` + `dreamcontext app install` is ONLY needed for native/Rust changes (lib.rs, NSPanel code). Observed and enforced across multiple sessions; bookmark bm_qxziL8tw. Source: file:dashboard/src/components/sleepy/AgentTerminal.tsx; changelog:2026-06-28|desktop.
- **App.tsx switch(nav.page) unmounts the active page — session-persistent components must hoist above the switch (2026-06-28)**: App.tsx returns exactly one page via `switch (nav.page)`. Navigating away unmounts the active page component, destroying all its state (WebSocket, PTY session, xterm instance, scrollback). Any component that must survive navigation (e.g., an embedded terminal) CANNOT live inside a page returned by the switch. Pattern: hoist the instance into a singleton mounted once at the app shell (outside the switch), toggle visibility with CSS (display:none), re-fit/re-attach on show. The xterm/PTY session stays alive; only the DOM visibility changes. Tear down only on explicit close/restart or app quit. Source: file:dashboard/src/components/sleepy/AgentTerminal.tsx; file:dashboard/src/App.tsx; changelog:2026-06-28|desktop.
- **Board view preferences: two-level shared+local persistence, never localStorage (2026-06-28)**: Tasks board view configurations (saved views with their filter/sort/group combination) are split across two server-side JSON blobs: `overrides/board.json` (version-controlled, "save for all" — team's canonical view definitions, permanent and shareable) and `state/board.local.json` (gitignored, "save for yourself" — per-machine overrides and private views). The client merges both on GET /api/board read (local overrides win on conflict; local-only views are appended). Neither file is subject to localStorage — the desktop app's per-launch loopback port resets localStorage on every relaunch. This is an extension of the server-side JSON pattern and adds the team-default + personal-override semantics. Apply this two-level split for any future UI config that needs to be both shareable and individually overridable. Source: file:src/server/routes/board.ts; changelog:2026-06-28|dashboard.

- **ask flag: agent must NEVER invent a value — ask the user first, or leave unset in no-user contexts (2026-06-23)**: `ask:true` on a CustomFieldDef signals human-judgment fields (e.g. time estimates) that the agent should not autonomously infer. The `renderOverrideBriefing()` ASK-FIRST rule is explicit: when a user is present, the agent must ask before creating the task and wait for the answer. In no-user contexts (sleep reconcile, autonomous runs), the field is left unset with a note — never filled with a guessed value. This is orthogonal to `required`: a field can be both `required` and `ask`, meaning the agent refuses to complete the task without a value but must get that value from the user, not invent it. Do NOT satisfy a `required` gate by inferring an `ask`-flagged value. Source: changelog:2026-06-23|tasks; file:src/lib/overrides.ts; knowledge:decisions/decision-task-format-override-and-custom-fields.
- **Required custom fields HARD-FAIL CLI actions; dashboard create path unaffected (2026-06-23)**: Required custom fields cause exit 1 at `tasks create`, `tasks complete`, and `tasks advance` in the CLI. The escape hatch is `--allow-missing-required` flag or the `DREAMCONTEXT_ALLOW_MISSING_REQUIRED=1` env var, intended for draft creation only. The dashboard create-task path goes through the server route (`POST /api/tasks`), which is NOT subject to the hard fail — only the CLI command layer enforces this. Do NOT add the hard fail to the server route; the asymmetry is intentional (agents use the CLI; the dashboard UI handles validation visually). Source: changelog:2026-06-23|tasks; file:src/cli/commands/tasks.ts.
- **Agent custom-field fill is guaranteed by TWO mechanisms; advisory briefing alone is insufficient (2026-06-23)**: The behavioral lever that makes an agent fill required custom fields is two-fold: (1) `renderOverrideBriefing()` injected into the main agent prompt AND every sub-agent at session start, and (2) the snapshot Active-Tasks block surfacing each task's custom-field line (including `⚠ UNSET (required)` warnings). Advisory briefing alone is insufficient — a stale agent session never reloads it after it was first injected, so required fields can remain unset across long sessions. The CLI-level hard fail (`exit 1`) is the deterministic backstop: it fires regardless of session age or briefing staleness, forcing the agent or user to set the field before completing the action. Do NOT remove the CLI hard fail on the grounds that briefing "should be enough." Source: changelog:2026-06-23|tasks; file:src/cli/commands/tasks.ts; file:src/lib/task-backend/types.ts.
- **Version model: code version (package.json) vs planning version (.active-version.json) are distinct (2026-06-23)**: The repo version (package.json, tauri.conf.json, Cargo.toml) tracks what is SHIPPED; the active planning version (`.active-version.json`, managed by `dreamcontext core releases active`) tracks the NEXT FEATURE GATE milestone. These numbers are independent. E.g., code shipped in v0.9.2; v0.10.0 is only the active planning milestone — feature docs labeled "v0.10.0" refer to the milestone, not the shipped code. Docs that label work with the planning version number are WRONG and must be corrected to the version the code actually shipped in. Source: changelog:2026-06-23|docs; file:DEEP-DIVE.md.
- **Merge convention: --no-ff PR-numbered merge commits + immediate branch deletion (2026-06-23, recurring)**: PRs are merged to main via `git merge --no-ff` with a commit message of `merge PR #NN: <desc>` (2 parents). This is the universal merge convention; never squash-merge or fast-forward-merge PRs. Source: commit:182e6a3; commit:83d3a2f; commit:d1dbb31; commit:f20d4d2.
- **Version-conflict resolution: always keep the HIGHER version (2026-06-23)**: When a feature branch and a release commit on main both bumped the version, resolve conflicts in version files (package.json, tauri.conf.json, Cargo.toml/Lock) to the HIGHER version. The branch's next-release number supersedes the interim release — e.g., branch bumped to 0.9.2 while main released 0.9.1: resolve to 0.9.2. Do not lose the interim release: mark it released in RELEASES.json before merging. Source: commit:182e6a3; changelog:2026-06-23|dashboard.
- **Overlapping-PR local integration strategy (2026-06-23)**: When multiple open PRs touch overlapping files, merge locally in conflict-ascending order rather than via the GitHub UI. Protocol: (1) stash any local uncommitted WIP to preserve it; (2) merge PRs in order of increasing overlap (lowest-conflict branch first); (3) resolve any conflicts — when both sides added a helper at the same location, keep both and add a separating closing brace; (4) confirm tests green; (5) fast-forward main and push (auto-closes PRs on GitHub). Check stash diff against the merged result before dropping the stash — the local WIP may be an earlier draft that the merged PRs fully supersede. Source: changelog:2026-06-23|tasks; commit:83d3a2f.
