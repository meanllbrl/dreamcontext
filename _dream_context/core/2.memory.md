- **Skill docs are edited ONLY in `skill/` (source of truth) and verified with `git status`; marker tests pin must-exist doc content (2026-07-06, ★★ recurring — 2nd occurrence class)**: The v0.11.0 Lab skill docs were silently lost because edits hit the INSTALLED copies (`.claude/skills/`) which the next `update` clobbered — while the task changelog claimed "skill docs updated." Same failure class as the earlier v0.11.0 doc loss. Consequence: agents misrouted "create insight" to `knowledge create`. Class-level defense now in place: (1) always edit `skill/SKILL.md` + `skill/references/*.md` and confirm the diff shows in `git status` before claiming docs shipped; (2) marker tests (`tests/unit/taxonomy-markers.test.ts`) pin must-exist doc content (Lab capabilities row, Entity Router section) so a clobber fails CI instead of silently regressing. This is the repo's standard defense for any doc content that MUST exist. Source: bookmark 2026-07-06; changelog:2026-07-06|skill; task:feat-lab-analytics-insights-subsystem.
- **Count caps are not size caps; clamp derived-index render state where it is READ (2026-07-06, ★★ review lesson)**: Two reusable findings from the Lab adversarial review. (1) A bounded list of unbounded strings still bloats — `HISTORY_MAX=50` capped entry COUNT while each entry's error string was unbounded; cap both (errors truncated to 300 chars). Apply to any ring buffer/history/log structure: bound count AND per-entry size. (2) Render-time state that indexes into a derived array (`hoverIdx` into `xKeys`) must be clamped at the point it is READ, not only where it is set — props can shrink between renders, leaving a stale index out of range. Source: bookmark 2026-07-06; file:src/lib/lab/sync.ts; file:dashboard/src/components/lab/LineChart.tsx; changelog:2026-07-06|lab.
- **Brain lives in its own GitHub repo, separate from the code repo; post-sleep auto git-sync is agent-merged, not git-merged (2026-07-04, decision by Mehmet)**: For cross-machine/multi-person brain collaboration, the dreamcontext brain (`_dream_context/`) will live in a dedicated GitHub repo that points at — not embeds — the paired code repo: separates access control (share the brain without the code, or vice versa) and keeps automatic sleep commits out of the code repo's PR/history noise. D0 (first shippable) scope is READ-ONLY: GitHub device-flow OAuth login (PAT fallback, token in macOS keychain), list-and-clone accessible `dreamcontext-brain` repos (tagged via a GitHub topic or root marker file), guided clone of the paired code repo, then a LOCAL-ONLY index build (`.gitignore`d, rebuilt per machine, never pushed) — zero push risk. Phase 2 adds one-way write (commit+push at sleep end, halt-and-notify on conflict) gated behind a secrets/absolute-local-path scrub. Phase 3 is the hard part: an AGENT (not `git merge`) resolves same-day conflicting sleeps semantically — union changelogs, content-merge knowledge files, keep the more-advanced task status — plus a manual `/dream-sync`-style skill and a personal-layer namespacing decision (episodic `1.user.md` content is person-specific; reuse the existing multi-person `people[]` roster rather than inventing new per-person paths — never-pushed vs namespaced-and-portable is the open question). Plan validation (v3.2, 3 reviewers, 5 iterations, 21 findings) locked five implementation decisions: (1) git credentials via GIT_ASKPASS + 0600 tmp file, token NEVER in remote URL; (2) brain repos private-by-default, attach requires trust confirmation (prompt-injection channel); (3) scrub gate before EVERY brain commit, WARN blocks only in headless contexts; (4) --resume (attended redo) vs --continue (commit in-progress merge) split; (5) brainSync token resolver intentionally secrets-first (reverse of resolveGitHubToken's env-first). `task-backend/github.ts` (already shipped for issue sync) is reused for future issue-mapping, not rebuilt. Source: session:9516bd8c-891e-4122-9948-c09c45f78041; task:github-cloud-collaboration-brain-repo-sync.

- **WKWebView `navigator.clipboard.writeText` mangles non-ASCII — use hidden-textarea + `execCommand('copy')` for encoding-safe writes (2026-07-01, ★★ reusable footgun)**: In this packaged app's WKWebView, `navigator.clipboard.writeText()` re-decodes UTF-8 bytes as Mac Roman (ç→√ß, ğ→ƒü, →→‚Üí) — on-screen text is unaffected, only the clipboard write mangles. Fix: route copy through a hidden `<textarea>` + `document.execCommand('copy')` (the OS's native text-copy pipeline), which round-trips UTF-8 correctly; keep `navigator.clipboard` only as a last-resort fallback. Apply to any WKWebView copy-to-clipboard path that may carry non-ASCII text. Source: file:dashboard/src/components/sleepy/agentSession.ts; task:agent-terminal-rendering-readability-polish.
- **xterm: prefer the DOM renderer over WebGL for an embedded low-throughput terminal — the WebGL glyph atlas ignores font-smoothing (2026-07-01)**: `@xterm/addon-webgl` rasterizes its own GPU glyph atlas, so CSS `-webkit-font-smoothing` has no effect on it — edges read hard/'sharp'. The DOM renderer draws real text nodes and gets native macOS anti-aliasing, at the cost of GPU compositing — an acceptable trade for a low-throughput viewport like the Claude Code TUI (mostly renders visible rows, no gaming-terminal scrollback firehose). Supersedes the WebGL choice recorded for this same terminal in `knowledge/desktop-beta-tauri-multivault.md` (flagged stale for sleep-product). Prefer the DOM renderer whenever an embedded terminal's readability matters more than max frame-rate. Source: file:dashboard/src/components/sleepy/agentSession.ts; task:agent-terminal-rendering-readability-polish.
- **xterm draws selection via `selectionInactiveBackground` (a faint default) whenever the terminal isn't the focused DOM element — always set both selection colors (2026-07-01, ★★ reusable footgun)**: Setting only `selectionBackground` leaves selection nearly invisible the moment the terminal loses focus, because xterm silently swaps to its own faint `selectionInactiveBackground` default (invisible on a light background). Fix: always set BOTH `selectionBackground` and `selectionInactiveBackground` to the same solid, high-contrast color when customizing an xterm theme. Source: file:dashboard/src/components/sleepy/agentSession.ts; task:agent-terminal-rendering-readability-polish.
- **Path/size validation on any locally-resolved input from remote/sync data must gate BEFORE reading bytes (2026-06-30, ★★ reusable footgun)**: An adversarial review of the GitHub image-bridge caught a Critical: `resolveLocalImagePath` honored arbitrary absolute paths and buffered the whole file into memory before checking the size cap — reachable via a remote issue body written back into a task mirror. Fix pattern: (1) an allow-listed root-containment guard (`isInsideRoot`, lexical, same idiom as `safeChildPath`) rejects any resolved path escaping the vault root before it's used; (2) a single `resolveImageBytes` choke point stat-gates the size cap BEFORE the read, not after. Apply this ordering (containment-check → size-stat → read) to any file path that originates from synced/remote content, not just user-typed CLI input. Source: commit:0d64ca9; file:src/lib/task-backend/github-assets.ts; task:github-sync-upload-local-task-images-so-they-resolve-in-issues.
- **Desktop orphan-process leak: root-cause fix is a JS-side parent-death watchdog polled via `kill(pid,0)`, not Rust process-group teardown alone (2026-06-30)**: Dashboard-server child processes leaked (~25 observed) because the Tauri shell only reaped the server on `RunEvent::ExitRequested`, which never fires on force-quit/crash/dev-rebuild — the server gets reparented to launchd and holds its loopback port forever. Fix: `src/server/lifecycle.ts` `startParentDeathWatch()` records the parent PID at spawn (`DREAMCONTEXT_PARENT_PID`, falling back to `process.ppid`) and polls it via `process.kill(pid, 0)` (ESRCH ⇒ gone; deliberately not `process.ppid` polling, which Node may cache stale); the server self-exits on any parent-death path and reaps its own PTY children first (`node-pty` `claude` sessions `setsid` into their own session, so a Rust process-group kill can't reach them). Rust side is belt-and-suspenders: spawn the child in its own process group and send SIGTERM→(1.5s)→SIGKILL to the whole group on exit, replacing the previous uncatchable SIGKILL that prevented the server from ever running its own cleanup. Gated on `DREAMCONTEXT_DESKTOP=1` so a terminal-run dashboard stays user-managed. Source: commit:0d64ca9; file:src/server/lifecycle.ts; file:desktop/src-tauri/src/lib.rs; task:fix-desktop-orphaned-dashboard-server-processes-leak-on-window-close.
- **React window pointer-listener teardown mid-drag: stabilize callbacks in refs, gate cleanup to unmount (2026-06-30, ★★ reusable footgun)**: Registering `window` `pointermove`/`pointerup` listeners inside a drag-start handler and removing them in a React cleanup effect causes the listeners to be torn down mid-drag if any `setState` (e.g. from the first `pointermove`) triggers a re-render that re-runs the cleanup. `pointerup` is never received, leaving the cursor stuck in the `grabbing` state. Fix: hold the mutation/update callback in a ref; write a stable `endDrag` closure that reads the ref; set the listener-teardown cleanup to run only on unmount (empty dep array or ref-based guard). This is a general React/pointer-events footgun, distinct from the WKWebView `drop`-not-delivered constraint. Source: commit:f37c871; file:dashboard/src/components/tasks/TimelineGantt.tsx; session:73222aa1-358e-4d22-a98c-3fdcbf9ab467.
- **WKWebView DnD: `drop` not delivered to mid-drag-mounted targets; custom-MIME getData() stripped (2026-06-30, ★★ platform constraint)**: WKWebView delivers `dragover` to conditionally-rendered (mid-drag-mounted) drop targets but does NOT deliver `drop` to them. Additionally, `dataTransfer.getData()` returns empty for any custom MIME type on `drop` — only `text/plain` survives. Fix pattern: store the dragged payload in a React ref (set on `dragstart`); execute the action on the source element's `dragend` (always reliable), never on `drop`. Board DnD avoids this because it uses `text/plain` and always-mounted targets — neither holds for dynamic split zones. Apply this pattern for all future drag-to-action on conditionally-mounted targets in WKWebView. Source: commit:e0b7eca; changelog:2026-06-30|sleepy; task:feat-sleepy-agent-surface-ux-redesign.

A `> *` width:100% rule compounds the problem by deforming child widths. Apply whenever a performance `contain` boundary is in the ancestor chain and a visually-floating element needs to escape it. Source: commit:e0b7eca; file:dashboard/src/components/sleepy/AgentDock.tsx; changelog:2026-06-30|sleepy.

- **Sleep consolidation lock: `sleep_started_at` IS the lock; O_EXCL stamp closes the race; 30m stale TTL self-heals crashes (2026-06-29)**: `sleep_started_at` in `.sleep.json` is the primary consolidation lock — `inspectSleepLock()` centralizes read + 30m stale TTL. `sleep start` refuses a live lock (exit 1), auto-reclaims a stale one (crash recovery), and accepts `--force` for deliberate override. Hooks suppress under a live lock but fall through a stale one. The Tauri launcher returns 409 on a concurrent desktop Sleep. An atomic O_EXCL file stamp lock (`src/lib/file-lock.ts`, 60s stale TTL) closes the cross-process check-then-stamp race — 8/16-way parallel `sleep start` yields exactly 1 winner. WHY: without the stamp lock, two processes could both see `sleep_started_at` unset and both proceed, producing corrupted epoch boundaries and parallel specialist writes to the same core files. Source: commit:4c9a166; file:src/lib/file-lock.ts; file:src/cli/commands/sleep.ts; file:src/lib/sleep-consolidation.ts.

- **Virtual/dynamic filter tokens: `@`-prefix resolves against live state, not stored field values (2026-06-29)**: Filter UI selections that need to track live project state (e.g. "the active sprint", "any completed sprint") must be implemented as `@`-prefixed pseudo-tokens (`@current`, `@backlog`, `@completed`) that resolve at filter time against live data (active-version, released-version set) rather than storing a literal value. The `@` prefix prevents collision with any real stored field value. This pattern separates the filter's INTENT (e.g. "the current sprint") from the underlying DATA (the actual version string), so a saved view filtered by "@current" always follows the live active sprint without requiring an update. Apply this pattern whenever a filter option represents a dynamic concept rather than a fixed stored value. - **`build:dashboard` does NOT deploy to the running app — always run full `npm run build` (2026-06-17)**: `npm run build:dashboard` rebuilds only `dashboard/dist/`. Source: commit:d106c5c; task:issue-20-excalidraw-knowledge.

CHANGELOG entries load first and crowd out knowledge/feature docs. Workaround: BM25 fallback still fires; net impact is recall degraded on large corpora. Fix: raise cap or switch to a token-count-based trim; track in ecc-inspired-roadmap task.
- **knowledge_access not updated by recall path (2026-06-05)**: knowledge_access in .sleep.json is only bumped by explicit `dreamcontext knowledge touch <slug>` calls, never by the BM25 or Haiku recall path. Staleness tracking therefore misses implicit access via recall hooks. Fix: call bumpKnowledgeAccess() in recall.ts whenever a doc is returned as a top hit.
