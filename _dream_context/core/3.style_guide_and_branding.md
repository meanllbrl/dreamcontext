## Branding

- **Name**: dreamcontext (lowercase, one word). NOT "Sleepy" — "Sleepy" is only the Search/Ask surface name and the macOS notch mascot persona. The project, npm package, CLI, and wordmark all remain "dreamcontext". No rename.
- **Binary/command**: `dreamcontext`
- **npm package**: `dreamcontext`
- **Banner**: ASCII art rendered in `src/cli/index.ts` using chalk.cyan and chalk.blueBright. Shown only in interactive mode (no args). Not shown when commands are run directly.

## Logo & Visual Identity (2026-06-27)

- **Logo mark**: Violet folded-diamond "dream gem" — light-lavender rounded diamond body + central leaf/eye detail + right wing + outer ring + glow effect. Implemented as a scalable SVG component at `dashboard/src/components/brand/BrandMark.tsx` (uses `useId()` for SSR/de-dup safety). This single component drives: sidebar brand lockup, context-constellation center dot, favicon, and macOS app/Dock icon.
- **Wordmark**: Two-tone "dreamcontext" — "dream" in neutral/light, "**context**" in violet. Never repeat the wordmark outside the sidebar brand lockup in the app UI.
- **Primary color**: Violet (CSS custom property family `--color-violet-*`; matches the ClickUp/Deep-Violet palette from the dashboard design system).
- **Favicon**: `dashboard/public/favicon.svg` — SVG version of the dream-gem logo.
- **App icons**: `desktop/src-tauri/icons/` — full set regenerated from the dream-gem (icns, ico, png at all sizes, iOS AppIcon variants, Android mipmap-* densities).

## Sleepy — Mascot & Surface Name

- **Sleepy** is the assistant/mascot persona name AND the name of the Search/Ask surface in the dashboard. It deliberately reuses the same name as the sleep-debt "Sleepy" level and the macOS notch capture — all three refer to the same character concept.
- **Voice**: grounded, local, no tokens. Sleepy never calls an LLM mid-task; it composes answers from what's already in the brain. Summary line for Ask: "Grounded · Extractive · No tokens."
- **Ask answers are zero-token**: composed purely from top recall hits, no LLM call. Always extractive (verbatim snippets with inline citations), never generative.

## Dashboard App Layout

- **Header (3-zone canonical layout)**: sidebar-toggle (left) · vault name centered · theme-toggle + avatar (right). No wordmark or logo in the header — single brand lockup lives in the sidebar only.
- **Sidebar brand lockup**: BrandMark SVG + "dreamcontext" two-tone wordmark, once, at the top of the sidebar. When sidebar collapses the lockup hides; the toggle moves to the header's left zone via `useSidebarCollapse.ts`.
- **Sleepy as default landing page**: registered as first Workspace nav item. New installs land on the Search/Ask surface.
- **No page headlines**: Pages do NOT render an h1 or `.page-title` headline. The sidebar nav label is the page title. Page bodies begin directly with content (master-detail, toolbar, or card). Any future page added to the dashboard must follow this convention — no exceptions.

## CLI Color Palette (chalk)

- Success: `chalk.green('✓')`
- Error: `chalk.red('✗')`
- Info: `chalk.cyan('ℹ')`
- Emphasis: `chalk.cyan(value)` for highlighted names/values
- Dim/secondary: `chalk.dim(text)` for hints and secondary info
- Headers in tables: `chalk.bold(text)`
- Interactive prompt: `chalk.cyan('dreamcontext') + chalk.dim(' > ')`

## CLI Output Conventions

- **Tables**: `formatTable(headers, rows)` from `src/lib/format.ts`. Column separators use `─` (not `-`). Auto-sized columns.
- **Success messages**: `success('Message here.')` — always a full sentence, lowercase after the first word.
- **Error messages**: `error('File not found: filename')` — specific, includes the offending value.
- **Info messages**: `info('Next: Use ...')` — for hints and next-step guidance.
- **No emoji in code or file output**: The `format.ts` helpers use `✓`, `✗`, `ℹ` which are symbols, not emoji. Markdown content files should not contain emoji.

## Voice & Tone (CLI)

- Terse. No "Great job!" or "Successfully completed the operation you requested." Just: `✓ Task created: my-task.md`
- Error messages name the thing that failed: `✗ Feature not found: auth` not `✗ Operation failed`.
- Hints use "Next:" prefix: `ℹ Next: Use dreamcontext features create <name> to add features.`
- Interactive prompts (via @inquirer/prompts) use title case for the message label: `'What is this project?'` not `'what is this project?'`

## File Naming Conventions

- Source: `kebab-case.ts`
- Template files: match the final output filename (e.g., `0.soul.md`, `4.tech_stack.md`)
- Task files: slugified name from user input (`my-task-name.md`)
- Feature files: slugified name from user input (`user-authentication.md`)
- Knowledge files: slugified name from user input (`jwt-token-strategy.md`)

## Code Style

- TypeScript strict mode. No `any` except where unavoidable with third-party types (e.g., Commander action callbacks).
- Named exports only for lib modules. The main CLI entry uses a default-style `main()` function but doesn't export it.
- Async/await throughout. No callback-style code.
- Error handling: CLI commands catch errors and use `error(msg)` helper to print them. Never let unhandled rejections surface as raw stack traces to the user.
- Target ~200-300 lines per source file. Current files are all well within this range.
